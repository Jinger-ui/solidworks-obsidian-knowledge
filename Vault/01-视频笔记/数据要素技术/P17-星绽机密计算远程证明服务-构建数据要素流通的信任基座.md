## 详细笔记

> 以下内容基于数据要素流通与隐私计算技术体系撰写，对应 B 站分 P「星绽机密计算远程证明服务：构建数据要素流通的信任基座」。**非 UP 逐字转写**；实操步骤请对照视频与官方文档。

### 1. 远程证明必要性

数据要素流通中，数据使用方需确认：**计算环境未被篡改、运行的是预期代码、TEE 由真实硬件提供**。远程证明（Remote Attestation）提供可验证的信任链。

### 2. 证明流程

1. Enclave 生成**引用报告**（Quote），含度量值（MRENCLAVE/MRSIGNER）
2. 证明服务验证 Quote 签名（Intel IAS/DCAP、ARM PSA）
3. 比对**黄金度量**（预期代码哈希）
4. 颁发**策略令牌**，允许密钥释放或任务执行
5. 审计系统记录证明结果

### 3. 星绽服务（课程案例）

**星绽**机密计算远程证明服务：
- 统一对接多种 TEE（SGX、SEV、TrustZone）
- 提供策略驱动的证明验证 API
- 与 KMS 联动：证明通过才释放数据密钥
- 构建数据要素流通的**信任基座**

### 4. 信任链

硬件 RoT → 固件 → TEE OS → Enclave 应用 → 证明服务 → 业务策略

### 5. 失败处理

- 度量不匹配：拒绝任务、告警
- 证明过期：重新证明（TcbInfo 更新）
- 吊销列表（CRL）：阻止 compromised 平台

### 6. 考试/实践要点

- 画远程证明时序图（用户→Enclave→证明服务→KMS）
- 解释 MRENCLAVE 与 MRSIGNER 含义
- 说明证明服务在可信数据空间中的位置

### 7. 策略语言

证明策略可用 Rego/CEL 表达：仅 Intel SGX且版本≥X 且 MRENCLAVE∈白名单 才释放密钥。

### 8. 灾备

证明服务本身需高可用；多地域部署，根证书离线保管。

### 9. 与 PKI 集成

证明服务应与企业 PKI 对接：Enclave 代码签名证书由内部 CA 签发，形成统一信任锚而非仅依赖 CPU 厂商。
